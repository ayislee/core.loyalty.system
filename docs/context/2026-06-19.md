# Context Harian - 2026-06-19

Dokumen ini merangkum konteks pembicaraan dan pekerjaan lintas aplikasi sampai 19 Juni 2026. Pada chat baru, baca dokumen ini bersama `issue.md`, lalu verifikasi kembali kondisi source code dan database sebelum membuat perubahan.

## Peta Aplikasi

1. `core.loyalty.system` adalah backend loyalty sekaligus proxy ke marketplace.
2. `client.loyalty.system` adalah frontend customer loyalty.
3. `cms.loyalty.system` adalah frontend management loyalty.
4. `backend-mediacartz` adalah backend marketplace dan nilai `MARKETPLACE_CORE` pada environment `core.loyalty.system`.
5. `frontend-mediacartz` atau `frontend-mediacartz-react` adalah frontend marketplace untuk `backend-mediacartz`.

## Aturan Arsitektur

1. Komunikasi dari `client.loyalty.system` ke `backend-mediacartz` harus selalu melalui `core.loyalty.system`.
2. `backend-mediacartz` menjadi sumber transaksi marketplace, produk, toko, pengiriman, dan status transaksi.
3. `core.loyalty.system` menjadi sumber data loyalty, voucher member, point, serta pencatatan settlement transaksi yang berasal dari platform loyalty.
4. Jangan mengasumsikan semua aplikasi masih berjalan dalam mode dev pada chat berikutnya. Periksa proses dan port jika diperlukan.

## Keputusan Bisnis Transaksi dan Point

1. Transaksi MIDTRANS dengan pengiriman kurir belum dianggap selesai hanya karena pembayaran berhasil.
2. Setelah pembayaran kurir berhasil, alur status yang dibahas adalah `verified`, kemudian `progressing` selama pengiriman, dan `complete` setelah barang diterima.
3. Perubahan status transaksi perlu dicatat pada history `transaction_status`.
4. Customer dapat menekan `Selesaikan` untuk mengubah transaksi menjadi `complete`.
5. Customer dapat menekan `Complain` untuk diarahkan ke WhatsApp toko dengan pesan awal yang menyertakan `transaction_number`.
6. Status `customer_receive_confirmation_status = complained` tidak menyembunyikan tombol `Selesaikan` dan `Complain`.
7. Konfirmasi `Selesaikan` menggunakan dialog, bukan konfirmasi browser biasa.
8. Jika tidak ada konfirmasi atau complain dalam 24 jam setelah barang terkirim, transaksi dapat diselesaikan otomatis.
9. Review dan rating produk hanya boleh dilakukan saat transaksi berstatus `complete`.
10. Point diberikan setelah transaksi benar-benar `complete`.
11. Transaksi yang memakai voucher tetap mendapat point berdasarkan uang yang dibayarkan.
12. Ongkos kirim tidak dihitung sebagai dasar point. Dasar point adalah `transaction_total_amount - transaction_shipping_fee`, minimum nol.
13. Bukti pemberian point harus diperiksa pada ledger `core.loyalty.system.point_histories`, bukan hanya dari status transaksi marketplace.

## Keputusan Bisnis Voucher

1. Voucher berlaku untuk kombinasi company penerbit dan produk yang berhubungan.
2. Pada halaman `/order`, voucher yang tidak berhubungan dengan produk dalam keranjang tidak ditampilkan.
3. Voucher tidak boleh digunakan pada company selain company penerbitnya.
4. Voucher tipe `amount` mendukung potongan nominal tetap atau persentase.
5. Nilai pada `member_voucher` mengikuti nilai potongan master voucher, bukan lagi selalu `Env.get('POINT') * voucher.number_point`.
6. Voucher tipe `free_delivery` tidak menggunakan `discount_type` dan `discount_value`.
7. SKU pada voucher `free_delivery` boleh kosong. SKU kosong berarti berlaku untuk seluruh produk terpilih; jika diisi, minimal salah satu produk yang dibeli harus memiliki SKU tersebut.
8. Detail transaksi customer perlu menampilkan informasi bahwa transaksi menggunakan voucher.

## Pengiriman dan GoSend

1. Catatan kurir pada `/order` menggunakan field lintas aplikasi `transaction_delivery_note`.
2. Catatan diteruskan melalui `core.loyalty.system` ke `backend-mediacartz` dan dimasukkan ke payload booking GoSend sebagai `routes[0].destinationNote`.
3. Satu transaksi tidak boleh membuat booking kurir GoSend lebih dari satu kali secara default.
4. Booking menggunakan lock dan pemeriksaan existing order. Multiple booking hanya boleh jika konfigurasi khusus mengizinkannya.
5. Timeline pengiriman di frontend menyaring status duplikat dan status internal seperti duplicate, ignored, failed, atau unauthorized.
6. Untuk delivery method `PICKUP`, sumber tanggal adalah `delivery_expected_datetime`.
7. Field lama `pickup_date` tetap dapat diisi dari `delivery_expected_datetime` untuk kompatibilitas penyimpanan dan tampilan.
8. Pesan error lama `pickup_date is required` berasal dari `PickupDeliveryMethod`, bukan dari `FormTransactionRetail`.

## Logging Retail Marketplace

1. Payload request retail dicatat sebelum validator agar request yang ditolak validator tetap terlihat.
2. Middleware yang digunakan adalah `backend-mediacartz/app/Middleware/LogRetailTransactionPayload.js`.
3. Middleware mencatat `method`, `url`, `user_id`, dan `request.all()`.
4. Middleware diregistrasikan di `backend-mediacartz/start/kernel.js`.
5. Middleware dipasang sebelum `.validator("FormTransactionRetail")` pada:
   - `POST /transaction/retail`
   - `POST /transaction/retail/order`
6. Log lama di controller sudah dihapus agar tidak terjadi double log.

## Kompatibilitas Frontend Marketplace

1. `frontend-mediacartz-react` ditetapkan kompatibel mulai Node `16.20.0` dan npm `8`.
2. Proyek menggunakan CRA `react-scripts 4.0.3` dan webpack 4.
3. Wrapper `scripts/react-scripts-compatible.js` menambahkan `--openssl-legacy-provider` hanya untuk Node 17 ke atas.
4. Build menggunakan `--max-old-space-size=4096` melalui wrapper.
5. Build berhasil diverifikasi pada Node `18.19.0`.
6. Banyak lint warning lama masih ada, tetapi bukan penghalang build.

## Pembatasan Metode Pembayaran Loyalty ke MIDTRANS

Implementasi dilakukan pada 19 Juni 2026 berdasarkan `core.loyalty.system/issue.md`.

Keputusan bisnis:

1. Store tetap boleh memiliki `store.store_payment_method = [4,6]` untuk mendukung MIDTRANS dan pembayaran Merchant pada POS.
2. Checkout `client.loyalty.system` hanya boleh menampilkan dan memilih payment dengan `ms_payment_identifier = MIDTRANS`.
3. `MERCHANT_PAYMENT` tidak dihapus dari master atau konfigurasi store karena masih dipakai POS dan channel marketplace lain.

Perubahan backend loyalty:

1. `core.loyalty.system/app/Controllers/Http/MasterController.js`, method `payment`, sekarang memanggil `MARKETPLACE_CORE/list/ms_payment` dengan query `ms_payment_identifier=MIDTRANS`.
2. Response tetap difilter strict di `core.loyalty.system` dengan normalisasi `trim().toUpperCase()` dan exact match `MIDTRANS`.
3. Bentuk response endpoint `GET /api/v1/member/master/payment` tetap `{ status, data }`.
4. Jika MIDTRANS tidak tersedia, endpoint mengembalikan `status: true` dan `data: []`.
5. `core.loyalty.system/app/Controllers/Http/TransactionController.js` tidak diubah. Flow tersebut sudah memaksa `MARKETPLACE_CORE_PAYMENT_ID`, default `4`.

Perubahan frontend loyalty:

1. `client.loyalty.system/src/pages/Order/index.jsx` menambahkan normalisasi identifier dan helper exact match MIDTRANS.
2. Hasil `FnMSPayment` difilter menjadi MIDTRANS sebelum disimpan ke state.
3. Daftar payment store menggunakan gabungan syarat ID diizinkan store dan identifier exact MIDTRANS.
4. Pemilihan payment non-MIDTRANS ditolak.
5. `Calculate` hanya menerima ID dari `selectedPayment` MIDTRANS dan tidak lagi fallback ke `store_payment_method[0]`.
6. Kalkulasi ongkir GoSend tidak lagi mengambil payment type dari elemen pertama `store_payment_method`.
7. Jika MIDTRANS tidak tersedia, UI menampilkan kondisi metode pembayaran tidak tersedia dan proses checkout berhenti.
8. `PreOrder` tidak diubah karena tidak menyediakan pemilihan payment ulang; popup/token MIDTRANS tetap memakai flow lama.

Yang tidak berubah:

1. `backend-mediacartz`, endpoint generic `list/ms_payment`, konfigurasi store, dan flow POS.
2. Logic transaksi, voucher, ongkir, point, settlement, callback, dan token MIDTRANS.
3. Struktur database.

Verifikasi:

1. `node --check app/Controllers/Http/MasterController.js` berhasil.
2. `git diff --check` pada file backend dan frontend berhasil.
3. `npm run build` pada `client.loyalty.system` berhasil dengan warning lint lama yang tidak terkait perubahan ini.

## Rencana Sebelumnya: Laporan Penjualan dan Balance Company

Rencana laporan penjualan dan balance pernah dibahas, tetapi `issue.md` saat ini sudah diganti dengan scope pembatasan MIDTRANS. Rencana laporan ini belum dianggap selesai diimplementasikan dan tetap disimpan di context untuk pekerjaan berikutnya.

Latar belakang:

1. Pembayaran marketplace dari `client.loyalty.system` ditampung dahulu oleh platform loyalty.
2. Company atau toko kemudian mempunyai saldo/tagihan kepada platform loyalty.
3. Setiap transaksi yang berasal dari loyalty harus dicatat sebagai data finansial terstruktur di `core.loyalty.system`.

Rencana utama:

1. Tabel transaksi finansial ternormalisasi, contoh `marketplace_sales`.
2. Ledger mutasi balance, contoh `company_store_balance_mutations`.
3. Tabel saldo ringkas per company/toko, contoh `company_store_balances`.
4. Rekap penjualan harian, contoh `daily_company_sales_reports`.
5. Service khusus untuk normalisasi transaksi, mutasi saldo, rekap harian, dan idempotency berdasarkan `transaction_number`.
6. Job rekonsiliasi status transaksi dari `backend-mediacartz` melalui `core.loyalty.system`.
7. API CMS untuk laporan harian, balance, mutasi balance, dan daftar transaksi.
8. Halaman CMS untuk laporan dan balance per company/toko.

Keputusan yang masih perlu disepakati sebelum implementasi laporan:

1. Apakah ongkir menjadi bagian balance toko atau biaya kurir/platform yang dipisahkan.
2. Apakah voucher loyalty ditanggung platform atau company.
3. Formula platform fee atau komisi.
4. Waktu saldo menjadi available. Rekomendasi saat ini: pembayaran sukses masuk pending, lalu menjadi available saat transaksi `complete`.

## Pengembalian Voucher Loyalty Saat Transaksi Gagal

Implementasi dilakukan pada 19 Juni 2026 berdasarkan `core.loyalty.system/issue.md`.

Keputusan bisnis:

1. Voucher yang dikembalikan adalah `transaction.member_voucher_id`, bukan voucher marketplace `transaction.voucher_id`.
2. Return hanya dipicu saat transaksi retail berakhir `rejected` atau `failed`.
3. Status sementara dan kegagalan kurir yang masih dapat di-retry tidak mengembalikan voucher.
4. Return tidak mengembalikan poin pembelian voucher dan tidak memperpanjang `expire_date`.

Perubahan `core.loyalty.system`:

1. Migration `1781013000000_voucher_exchange_lifecycle_schema.js` menambahkan referensi transaksi, `store_id`, status `exchanged/returned`, timestamp, dan alasan return pada `voucher_exchanges`.
2. Unique key lama voucher/merchant diganti dengan unique key member voucher/referensi transaksi agar voucher returned dapat digunakan pada transaksi baru.
3. `redeem_merchant_id` dibuat nullable karena database development tidak memiliki master `redeem_merchants`; otorisasi tetap memvalidasi partner pemilik master voucher.
4. `MemberVoucherLifecycleService` menangani exchange dan return dalam database transaction dengan row lock dan idempotensi.
5. Return-before-exchange disimpan sebagai return intent, sehingga job exchange yang terlambat tidak mengubah voucher menjadi used lagi.
6. Endpoint baru: `POST /api/v1/merchant/member/voucher/return`, dilindungi `AuthMerchant`.
7. Endpoint exchange lama sekarang mewajibkan referensi transaksi dan memakai lifecycle service yang sama.
8. Trait soft delete dihapus dari `VoucherExchange` karena tabel tidak memiliki `deleted_at` dan histori lifecycle tidak boleh dihapus.

Perubahan `backend-mediacartz`:

1. Event exchange voucher dipindahkan setelah commit transaksi retail dan membawa `transaction_id` serta `transaction_number`.
2. `LoyaltyService.returnVoucher` memanggil endpoint return di core loyalty.
3. Queue exchange dan return memakai lima percobaan dengan exponential backoff serta mempertahankan failed job.
4. Worker sekarang gagal jika HTTP timeout/error atau core mengembalikan `status: false`.
5. `TransactionService.processReject` mengantrekan return setelah status lokal berhasil di-commit.
6. Cabang approval yang benar-benar menghasilkan status `failed` juga mengantrekan return.
7. Row lock dan guard status terminal mencegah callback reject paralel mengembalikan stok, voucher, atau counter lebih dari sekali.

Frontend:

1. Tidak ada endpoint return yang dipanggil dari browser.
2. `client.loyalty.system/src/pages/Order/index.jsx` sudah memuat ulang voucher aktif dari core setiap halaman `/order` dibuka, sehingga tidak memerlukan perubahan tambahan.

Verifikasi:

1. Migration berhasil dijalankan, di-rollback, dan dijalankan ulang dari schema lama.
2. Smoke test exchange, retry exchange, return, retry return, return-before-exchange, dan late exchange berhasil.
3. Hasil akhir smoke test menunjukkan `member_vouchers.used = 0` dan semua lifecycle terkait berstatus `returned`.
4. Fixture smoke test sudah dihapus dan tidak meninggalkan data `codex-test-*` atau transaksi `codex-*`.
5. Seluruh file JavaScript yang berubah lolos `node --check`.
6. Test backend `node ace test unit --files=utils.spec.js --bail` lulus 4 test.
